# Отчёт: баг геометрии «ТС едет по домам» — диагностика (сшивка ТС→shape)

Ветка: `feature/gtfs-freshness` · Дата: 2026-07-14 · Статус: **Вердикт G1.
Фикс реализован (Путь B, флаг `vehicle_direction_shape`), на staging, верифицирован
на живых данных. Merge в main + прод — по команде владельца. См. раздел «Фикс» ниже.**

## Симптом
На карте часть машин едет не по улице, а через кварталы/дома. Пример владельца:
на одном участке два автобуса (55 и 46), один по дороге, другой «по домам».

## Механизм (по коду)
Маркер живого ТС проецируется на geometry маршрута, но **shape выбирается только
по номеру линии, без учёта направления**:
- `home_map_screen.dart:780` — `path: _shapeCache[v.line]` (ключ = номер линии).
- `_ensureShapesFor` → `getShapeByLineNumber(line)` тянет **каноническое
  направление** (dir0, «голый» `route_id`).
- В `VehicleDto` направления нет — фронт физически не может выбрать нужное.

После F8 у линии два направления с разными shape (`00046` dir0 / `00046-1` dir1),
и терминалы часто разные (46: *Beograd na vodi→Mirijevo* vs *Mirijevo→TC
Galerija*). Машина, едущая по НЕканоническому направлению, проецируется на
полилинию канонического → её маркер «прилипает» к параллельной/другой улице.

## Вердикт: **G1 — неверный ключ сшивки ТС→shape** (корень)
Shape выбирается по номеру линии → всегда каноническое направление, без учёта
реального направления машины.
- **G2** (проекция на глоб.-ближайший сегмент) — вторична: оконная проекция (F1)
  не спасает, если сам shape чужой.
- **G3** (heading по чужому `all_stations`) — следствие: при наличии path фронт
  берёт heading из тангенса этого (неверного) shape (`headingAt`), поэтому и
  стрелка врёт. Бэкендовый heading из реального `all_stations` перекрывается.

## Пути фикса (жду выбор владельца)

- **A. Фронт: выбирать направление по позиции+heading.** Грузить ОБА shape линии,
  ститчить ТС к тому направлению, к чьей полилинии он ближе И чей тангенс
  согласуется с `v.heading` (heading уже направление-зависимый, из `all_stations`).
  С гистерезисом (запомнить выбранное направление в треке, не дёргать на
  перекрывающихся участках). **Плюс:** без изменения API/контракта, быстрее.
  **Минус:** возможен flip-flop на общих участках — лечится гистерезисом+heading.
- **B. Бэкенд: резолвить направление и отдавать `route_id` направления.** Бэкенд
  уже парсит `all_stations` (для heading) — сопоставить его с двумя GTFS-
  направлениями (по конечным/последовательности остановок) и вернуть точный
  `route_id` (напр. `00053-1`) в `VehicleDto`. Фронт ститчит по `v.route_id`.
  **Плюс:** авторитетно, переиспользует F8-shape. **Минус:** +логика матчинга на
  бэке, +1 поле в контракте.
- **C. Отдавать `all_stations` целиком** и проецировать на реальную геометрию
  рейса. Точнее всего, но станционный полилайн грубее shape, крупнее payload,
  расходится с GTFS-рендером остального.

**Рекомендация:** **A** как основной (фронт-only, без контракта, чинит корень;
heading даёт надёжный disambiguation), с гистерезисом. **B** — если A даст
дрожание. За фиче-флагом (сравнить старое/новое). Регрессионный тест: ТС dir1
проецируется на СВОЙ shape, не на канонический.

## Фикс (2026-07-14, Путь B, за флагом `vehicle_direction_shape`)

Реализовано по решению владельца — **бэкенд резолвит направление** (детерминированно
из `all_stations`), фронт ститчит по нему.

- **Резолвер** `backend/src/lib/direction.ts` (чистая функция): матчит конечные
  точки рейса (`all_stations` origin→destination) с концами каждого направления
  линии. При неоднозначности/пустом рейсе/одном направлении → `null` (фолбэк на
  каноническое, без краша).
- **Концы маршрута — в `lines.json`** (`build-gtfs.mjs` добавляет `origin_lat/lon`,
  `dest_lat/lon`), поэтому резолв не грузит shapes на каждый запрос (важно для
  nearby с фан-аутом по 12 остановкам).
- **Контракт (аддитивно):** `ArrivalDto.direction_route_id` +
  `VehicleDto.route_id` = route_id реального направления (фолбэк канонический).
  `route_id` в arrivals остался каноническим.
- **Фронт** (`home_map_screen`): при флаге ON ститчит по `v.routeId`
  (`getShapeByRouteId`), OFF → по номеру линии (как раньше). Нет route_id →
  straight-line ease (безопасная деградация).
- **Флаг** `vehicle_direction_shape`: OFF на проде, ON на staging.

**Верификация (staging, живые данные):** `/vehicles/nearby` теперь несёт
`route_id`; на выборке машин на двунаправленных линиях **4/4** `route_id` указывает
на направление, чей shape ближе к GPS (29N→`50029-1`, 27N→`50027-1`, 72N→`50072`,
26N→`50026`). Регрессионный тест — эталон линии 53/P21548 (`test/direction.test.ts`).
Тесты: backend 64/64, flutter 121/121, tsc/analyze чисто.

**Как проверить (без терминала):** на **https://preview-gtfs-freshness.stigla.pages.dev**
(флаг ON) машины на линиях с расходящимися направлениями (26, 23, 46, 55, 95…)
едут по своей улице, а не через кварталы. Сравнить с продом (флаг OFF). При
дневном трафике разница заметнее (сейчас межпик — мало машин).
