## H0 — какой кран (факт, без секретов в тексте)

Фактический upstream: система **`online.bgnaplata.rs`**, метод
**`/sr/announcement_arrival/`** (POST, form-urlencoded, поля `r`=stop_id,
`b`=клиентский id, плюс фикс-поля из `TRANSIT_SOURCE_FORM_EXTRA_JSON`).
Это **не** `publicapi/v3` / `data_bulletin`, а веб-эндпоинт «объявления
прибытий» той же системы.

Важно: по **содержимому** этот метод даёт корректные публичные данные —
`line_number` публичный, `station_name` совпадает с GTFS, внутри `all_stations`
первый элемент несёт наш же `stop_id`. Пример ответа для `21047` (Зелени Венац):
`line_number:"71"`, `station_name:"Zeleni venac"`, `actual_line_number:"5094"`
(внутренний код, мы его не используем). Значит смена метода на `data_bulletin`
**не является** причиной расхождения линий — переключать его ради этой задачи
смысла нет (отдельное решение, если нужно по другим причинам).

## H3 — совпадает ли id-пространство остановок (главный подозреваемый — снят)

Наш GTFS `stop_id` уходит в upstream как `r` напрямую
([arrivals.ts:15-18](backend/src/lib/arrivals.ts:15) →
[transitProvider.ts:28-30](backend/src/lib/transitProvider.ts:28)). Проверил,
резолвится ли он у источника в ту же остановку: в ответе есть `station_name` и
`all_stations[].id`.

**Результат: 76/76 совпадений имени станции.** Ни одного случая, где `r=<наш

(Это же снимает подозрение F1 про «плывёт не там» на почве чужой полилинии — по

### Усилитель на стороне приложения (независимо от фида)

Экран остановки показывает состав как **объединение** GTFS-набора и live-набора
([stop_screen.dart:150-155](app/lib/presentation/screens/stop_screen.dart:150)):
```dart
final arrivingLines = board.arrivals.map((a) => a.line).toSet();
final allLines = {...?stopLocation?.lines, ...arrivingLines} ...
```
Из-за этого:
- новые живые линии **подмешиваются** к GTFS-набору (по факту это спасает — иначе
  их бы не было видно совсем), но
- устаревшие GTFS-«призраки» (линии, которые сеть уже сняла/переименовала)
  **остаются висеть**, даже если их нет ни в живом потоке, ни у эталонов. Это и
  есть вероятный источник исходной жалобы «у нас есть лишние линии». Живой поток
  тут точнее справочника.
