# Задача: чистка фиче-флагов — документировать, прибрать dead-code, предложить сокращение

Ветка: `fix/flags-cleanup` (новая, от свежего main; можно продолжить в
`fix/flags-audit`, если worktree жив) · Отчёт:
`docs/reports/2026-07-14-flags-cleanup.md` (RU)

## Контекст

Аудит (отчёт `2026-07-14-flags-audit.md`) показал: прод здоров, рассинхронов нет, но
флагов накопилось 12 — владелец уже не помнит, кто за что и зачем. Проблема не в
состоянии (оно верное), а в **гигиене и памяти**. Задача — почистить в трёх
измерениях: документировать (чтобы «не помню» больше не случалось), прибрать
dead-code, и **предложить** сокращение флагов у стабильных фич — но убирать флаги
**только по явному решению владельца по каждому**.

## Что сделать

### 1. Реестр флагов (главное — лечит «не помню кто за что»)
Заведи живой документ `docs/FEATURE_FLAGS.md` (или расширь существующий
`docs/feature-flags.md`, если он есть) — единый реестр, поддерживаемый впредь.
Таблица по каждому флагу:
`флаг | что гейтит (фича/экран) | прод | staging | введён (дата/задача) | статус (постоянный за-флагом / временный раскаточный / кандидат на удаление)`.
Плюс короткое правило поддержания: новый флаг → строка в реестре; фича стабильна и
флаг больше не нужен → удалить и здесь. Внеси в DoD (CLAUDE.md), чтобы реестр не
протухал, как и галерея.

### 2. Прибрать dead-code (точно, без вопросов)
- Удалить `scheduleFallbackEnabledProvider` (`providers.dart:90`) — нигде не
  watch-ается, клиент читает `.scheduleFallback` напрямую. Рудимент.
- Проверить, нет ли других мёртвых флаг-провайдеров/геттеров (объявлены, не
  используются) — убрать.
- Прибрать мусорные `geocode:*` записи в прод-KV (не флаги, гигиена).

### 3. Явные дефолты в KV (задокументировать намерение)
Три флага в проде OFF по неявному дефолту (нет в KV): `analytics_show`,
`coverage_map_show`, `coverage_on_main_map`. Выставить их явно `=0` в прод-KV —
эффект не меняется, но намерение «осознанно выключено» становится
зафиксированным (а не «забыли завести»).

### 4. Предложить сокращение флагов — [STOP & ASK], решение по каждому за владельцем
Раздели 12 флагов на категории и **предложи** (не выполняй):
- **Постоянные за-флагом** — фича живёт за флагом по дизайну (напр. `analytics_show`
  — черновой UI; `coverage_*` — эксперимент). Флаг оставить.
- **Временные раскаточные, ставшие стабильными** — фича в проде ON, стабильна,
  откат уже не нужен → флаг можно удалить, впаяв рабочий путь. Кандидаты (проверь по
  дате/стабильности): `symbol_layer`, `timed_trajectory`, `vehicle_direction_shape`,
  `live_position_only`, `schedule_fallback`, `schedule_map` — те, что раскатаны и
  давно/стабильно ON.
- **Свежие (сегодняшние)** — `nearby_list`, `nearby_sort_board`, свежие движения —
  НЕ трогать, пусть поживут ради отката.
Для каждого кандидата на удаление дай: с какого времени ON, насколько стабилен, что
теряется при удалении флага (мгновенный откат), рекомендация (убрать/оставить).
**[STOP & ASK] — таблица кандидатов, владелец решает по каждому «убрать/оставить».
Ни один флаг не удаляется без явного да.**

## Границы
- Пункты 1-3 (реестр, dead-code, явные дефолты) — сделать, они безопасны.
- Пункт 4 (удаление флагов) — **только предложить**, выполнять по решению владельца
  по каждому флагу отдельно.
- Состояние прод-флагов (что ON/OFF) не менять сверх явных дефолтов п.3.
- Код EN, отчёт RU, без секретов.
- Merge — по команде владельца.
