# Задача: плавное движение ТС через timed-траекторию (as-of время + план вперёд)

Ветка: `feature/timed-trajectory` · Worktree: `../stigla-timed-trajectory`
Отчёт: `docs/reports/2026-07-XX-timed-trajectory.md` (RU)

> Замечание по контексту для сессии: уже есть ветка `feature/dead-reckoning`
> (маркеры продолжают движение между фиксами, бэкенд терпит устаревший кэш без
> `captured_at`) — НЕ влита в main, отчёт внутри ветки. Эта задача её замещает и
> обобщает: не «клиентский аниматор гадает скорость», а «сервер отдаёт
> тайминговый план вперёд, клиент его проигрывает». Перед стартом ознакомься с
> `feature/dead-reckoning` (git log/diff) и переиспользуй, что применимо; старую
> ветку не мержим, её судьба — после этой задачи (вероятно закрыть).

## Контекст и мотивация

Наш аниматор (`app/lib/core/vehicle_track_animator.dart`) сознательно
консервативен: маркер никогда не двигается за последний известный фикс, отсюда
поведение «стоит → прыжок» и потеря плавности против конкурентов.

Продуктовое исследование (эталоны плавности) показало общий приём: плавность
берётся НЕ от частой отправки сырых координат, а от **тайминговой траектории
вперёд** — на каждый ТС даётся опорное «as-of время» (≈сейчас) и
последовательность сегментов пути с абсолютным временем начала и длительностью;
клиент плавно анимирует ТС по этому плану. Частый опрос при этом не нужен —
план обновляется редко, анимация идёт локально. Это снимает и наш соблазн
учащать опрос (остаёмся на 30с и на общем SWR-кэше — нагрузка на upstream не
растёт).

Цель: дать нашему ТС плавное движение-вперёд по маршрутной полилинии,
управляемое временем, а не «прыжок к фиксу». Rendering и источник не меняем по
сути — меняем модель того, ЧТО анимируется.

Код/комментарии — EN, отчёт — RU, публичный текст без секретов и upstream-
деталей (URL/параметры источника — только в env, как всегда).

## Фаза 0 — диагностика upstream (ОБЯЗАТЕЛЬНО ПЕРВОЙ, [STOP & ASK])

Прежде чем строить модель — выяснить, что реально есть в ответе источника.
В `backend/` (`src/lib/transitProvider.ts` и сырой ответ провайдера) определить:

1. **Есть ли у позиции ТС время фикса** (timestamp/возраст «снято N секунд
   назад»), или отдаётся только координата «сейчас» без времени?
2. Есть ли что-то, кроме точки: скорость, курс от источника, признак
   «на остановке / в движении»?
3. Как часто реально меняется координата конкретного ТС между нашими опросами
   (по логам/наблюдению) — чтобы понять типичный шаг фикса.

Вывести это в отчёт как «Фаза 0». **[STOP & ASK]:** по результату выбрать модель
и согласовать с владельцем, ПРЕЖДЕ чем реализовывать фазу 1:

- **Сценарий A — upstream даёт время фикса.** Строим полную модель: бэкенд
  прокидывает «as-of время» в клиент; клиент строит план вперёд по маршрутной
  полилинии от спроецированной позиции и анимирует по времени. Наиболее близко
  к эталону, минимальная ошибка.
- **Сценарий B — времени нет.** «as-of время» = момент получения ответа;
  экстраполяция вперёд по оценке скорости (из двух последних фиксов вдоль
  маршрута). Работает, но копит ошибку — обязательна мягкая коррекция на новый
  фикс (без рывка назад).

Не реализовывать обе — по STOP & ASK делаем выбранную.

## Фаза 1 — реализация выбранной модели

Общее для A и B:

### Бэкенд (`backend/`)
- Обогатить ответ по ТС полем «as-of время» (реальное из upstream в A;
  серверное время ответа в B) — **аддитивно**, не ломая текущий контракт
  (клиенты старой версии игнорируют новое поле). Задокументировать поле.
- Никаких изменений политики 30s-кеша / SWR / fan-out. Источник опрашивается
  как раньше.

### Клиент (`app/`)
- **Модель timed-траектории** в чистом, тестируемом виде (расширить или заменить
  `vehicle_track_animator.dart`): по текущей позиции ТС + маршрутной полилинии
  (`vehicle_route.dart` / `route_path.dart`, проекция уже есть — переиспользовать
  оконную проекцию из F1) построить план движения вперёд; анимировать маркер по
  **времени** (as-of → сейчас), а не по «доезжай до последнего фикса и стой».
- **Движение только вперёд, без рывка назад** при коррекции: приход нового фикса
  не телепортирует и не откатывает — мягко подтягивает план (как у эталонов:
  «вперёд, редкие короткие паузы, без дёрганья назад»).
- **Согласованность с idle=0 (термика).** Не сломать прошлый фикс перегрева:
  тикер анимации живёт, только пока ТС реально движется по плану; доехал до
  конца плана и ждёт свежих данных — тикер и сэмплер останавливаются
  (`hasPendingMotion == false`), в фоне/на скрытой вкладке — пауза. Плавность
  добавляем, но покой остаётся нулевым по кадрам.
- **«Застрял».** Пересмотреть staleness-эвристику: теперь «доехал до конца
  плана» ≠ «застрял». Застрявший — это когда план исчерпан И свежих фиксов нет
  дольше порога. Не регрессировать grace-period.
- То же поведение для карты подъезжающих к остановке (`live_vehicles_map.dart`),
  чтобы оба экрана двигали ТС одинаково.

### Фиче-флаг
- Всё за флагом (напр. `timed_trajectory`): OFF на проде, ON на staging.
  При OFF — текущее консервативное поведение (маркер не идёт за фикс).
  Деградация: если план построить нельзя (нет полилинии/данных) — тихий фолбэк
  на старый аниматор, без визуального сбоя.

## Definition of Done (CLAUDE.md, 6 пунктов)

1. Код в ветке; тесты зелёные. Новые юнит-тесты на чистую модель траектории:
   построение плана, анимация по времени, коррекция-без-отката, отсутствие
   движения за пределы плана, `hasPendingMotion` в покое = false. Бэкенд —
   тест на аддитивное поле (старый контракт не сломан).
2. Отчёт RU: Фаза 0 (что нашли в upstream) + какой сценарий выбран и почему +
   реализация. Замер «до/после» — по возможности на staging-preview глазами
   (плавность, отсутствие отката, покой = 0 кадров).
3. BACKLOG: отметить (было в «в работе» как dead-reckoning) → обобщено в
   timed-trajectory; судьба старой ветки `feature/dead-reckoning`.
4. Видимого нового UI-состояния нет (то же движение, плавнее) — если появится
   заметное состояние, обновить галерею; иначе не трогать.
5. Раздел «Как проверить» без терминала: сессия сама деплоит staging-preview
   (per-branch worker version + Pages-превью по `docs/staging.md`); владельцу —
   URL, доступы, что смотреть (ТС едет плавно вперёд, при обновлении не
   откатывается назад, стоящие ТС не жгут кадры/батарею).
6. Merge в main — по явной команде владельца.

## Что НЕ делаем
- Не учащаем опрос upstream (плавность — от timed-траектории, не от частоты).
- Не меняем рендерер/DPR, не трогаем SWR/кэш/fan-out.
- Не тащим чужой формат данных как кальку — своя модель под наши shapes.
